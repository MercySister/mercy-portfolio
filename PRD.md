## 2. 产品概述

Mercy Portfolio 是一个面向招聘方、行业同行及潜在合作方的个人简历网站。产品以票券、档案和系统状态为核心视觉语言，通过结构化内容展示个人简介、工作经历、代表项目、教育经历和兴趣，并提供访问统计、点赞及访客留言能力。

网站默认使用中文，支持切换为英文。前端为单页应用，访问量、点赞量和留言由独立 API 与 PostgreSQL 数据库承载。

## 3. 产品目标

### 3.1 核心目标

1. 让访问者快速理解 Mercy Mu 的职业定位、核心能力与主要履历。
2. 通过代表项目呈现 AI、汽车及复杂系统产品的实际经验。
3. 同时服务中文和英文访问者。
4. 通过点赞和留言提供低门槛互动方式。
5. 支持脱离 Replit 后部署到个人域名。

### 3.2 非目标

当前版本不包含：

- 内容管理后台；
- 用户注册、登录或个人账户；
- 简历 PDF 下载；
- 博客、文章或动态发布；
- 留言公开展示与访客间互动；
- 管理端数据看板；
- 多语言内容自动翻译。

## 4. 目标用户与场景

### 4.1 目标用户

- 招聘方或用人经理：快速判断候选人的经历、能力和岗位匹配度。
- 行业同行或合作方：了解代表项目、专业方向和可合作领域。
- 海外访问者：通过英文内容了解完整履历。
- 网站所有者：通过访问量、点赞和邮件通知感知网站互动情况。

### 4.2 核心使用场景

1. 访问者打开个人域名，首先看到姓名、职业定位和内容导航。
2. 访问者切换不同档案页签，查看经历、项目、教育和兴趣。
3. 英文访问者切换至英文，当前页面内容同步切换。
4. 访问者对内容点赞，并看到累计访问量和点赞量。
5. 访问者填写姓名和留言，提交后收到成功反馈；网站所有者可收到邮件通知。

## 5. 信息架构

网站仅包含一个公开页面，默认路由为 `/`。

| 一级区域 | 主要内容 |
| --- | --- |
| Hero | 姓名、产品经理定位、网站状态、档案导航 |
| 关于 | 基本信息、核心理念、核心能力 |
| 经历 | 公司、任职时间、角色、标签、职责与成果 |
| 项目 | 车载 AI 智能体、车载机器人代表项目 |
| 教育 | 学校、专业、学术方向、校园经历 |
| 兴趣 | 泰拳、滑雪、攀岩、跳舞、徒步及活动策划经历 |
| 固定底栏 | 留言入口、语言切换、访问量、点赞量 |
| 留言弹窗 | 访客姓名、留言内容、提交状态与成功反馈 |

## 6. 功能需求

### FR-01 首页与视觉框架

产品应展示统一的票券式视觉框架，包括动态波形、条码、虚线分隔、状态标签、左右状态条和顶部滚动进度条。

验收标准：

- 访问 `/` 后显示 Hero 和默认“关于”内容；
- 页面滚动时顶部进度条同步更新；
- 页面背景及 Hero 波形保持动态效果；
- 未匹配路由显示 404 页面。

### FR-02 内容导航

Hero 区域应提供以下五个页签：关于、经历、项目、教育、兴趣。

验收标准：

- 默认选中“关于”；
- 点击页签后，仅展示对应内容；
- 当前选中页签具有明确的视觉状态；
- 内容切换包含淡入淡出或位移动画；
- 页签切换不触发整页刷新。

### FR-03 中英文切换

固定底栏应提供 CN 和 EN 两种语言选项。

验收标准：

- 默认语言为中文；
- 点击 EN 后，导航、简介、经历、项目、教育、兴趣和留言弹窗文案切换为英文；
- 点击 CN 后恢复中文；
- 切换语言时保留当前选中的内容页签；
- 当前版本不持久化语言选择，刷新后恢复中文。

### FR-04 关于信息

“关于”页签应展示：

- 个人基本信息；
- 挑战型工作思维及职业摘要；
- 0 to 1 Builder、Agile、AI Native 等能力标签；
- 商业研判、全链路项目、敏捷抗压、持续学习等核心能力。

隐私要求：

- 姓名、手机号和邮箱在页面上应以掩码形式展示；
- 当前实现仅为界面掩码，原始联系方式仍存在于前端源码中，正式上线前应确认是否接受该公开风险。

### FR-05 工作经历

“经历”页签应以时间和公司为单位展示工作经历。每段经历包含：

- 公司与任职时间；
- 职位与状态；
- 业务或能力标签；
- 分模块的职责、项目和结果说明。

当前内容覆盖阿维塔、达能亚太、喜茶及 Cleen/可印等经历。

### FR-06 代表项目

“项目”页签应展示代表项目的背景、职责和结果。

当前项目范围：

1. 车载 AI 智能体：角色设定、拟人化交互、情感陪伴与用户记忆。
2. 车载机器人：AI 与硬件结合、原子能力库和场景服务体系。

### FR-07 教育与兴趣

“教育”页签应展示学校、学院、专业、时间、学术方向和校园经历。

“兴趣”页签应展示运动兴趣卡片及活动策划相关说明。兴趣卡片支持悬停和点击动效，但当前不进入详情页。

### FR-08 访问统计

网站应在固定底栏展示累计访问量。

验收标准：

- 首页加载时读取当前统计数据；
- 页面组件首次挂载时记录一次访问；
- 访问记录成功后刷新统计数据；
- 数值首次出现时播放递增动画；
- 当前版本按页面加载次数计数，不做独立访客去重。

### FR-09 点赞

访问者应能在固定底栏为网站点赞。

验收标准：

- 未点赞时可点击点赞按钮；
- 点赞成功后刷新累计点赞量；
- 点赞成功后按钮显示已点赞状态并禁用重复点击；
- 浏览器通过 `localStorage` 中的 `mercy_liked` 标记限制重复点赞；
- 清除浏览器数据或更换浏览器后仍可再次点赞，服务端当前不做身份去重。

### FR-10 访客留言

访问者应能从固定底栏打开留言弹窗，填写姓名与留言内容。

验收标准：

- 点击“给我留言”后显示模态弹窗；
- 姓名和留言均为必填项；
- 点击遮罩或关闭按钮可关闭弹窗；
- 提交期间按钮进入禁用和加载状态；
- 提交成功后显示完成状态，并在约 2.5 秒后自动关闭；
- 留言写入数据库，包含 ID、姓名、内容和创建时间；
- 配置 SMTP 时异步发送通知邮件；
- 未配置 SMTP 或邮件发送失败时，不影响留言入库和接口成功响应。

### FR-11 健康检查

API 应提供 `GET /api/healthz`，正常运行时返回：

```json
{ "status": "ok" }
```

该接口用于部署平台的服务启动检查，不依赖业务数据读取。

## 7. 交互与状态要求

| 场景 | 当前行为 |
| --- | --- |
| 首次进入 | 中文、“关于”页签、读取统计并记录访问 |
| 切换页签 | 保持语言状态，播放内容切换动画 |
| 切换语言 | 保持页签状态，切换当前页面所有可翻译文案 |
| 点赞成功 | 写入本地标记、刷新统计、按钮变为已点赞 |
| 留言提交中 | 禁用提交按钮并显示发送中文案 |
| 留言成功 | 显示完成反馈，约 2.5 秒后关闭 |
| API 请求失败 | 当前缺少面向用户的明确错误提示 |

## 8. 数据与接口

### 8.1 数据模型

`site_stats`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | serial | 主键 |
| visitor_count | integer | 累计访问次数，默认 0 |
| like_count | integer | 累计点赞次数，默认 0 |

`messages`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | serial | 主键 |
| name | text | 留言人名称 |
| content | text | 留言内容 |
| created_at | timestamptz | 创建时间 |

### 8.2 API 清单

| 方法 | 路径 | 用途 | 成功状态 |
| --- | --- | --- | --- |
| GET | `/api/healthz` | 服务健康检查 | 200 |
| GET | `/api/stats` | 获取访问量与点赞量 | 200 |
| POST | `/api/visitors` | 访问量加一 | 200 |
| POST | `/api/likes` | 点赞量加一 | 200 |
| POST | `/api/messages` | 保存访客留言 | 201 |

OpenAPI 文件是接口和生成类型的唯一来源。修改接口后应重新运行 `pnpm run codegen`。

## 9. 响应式与兼容性

- 页面应支持主流现代桌面和移动浏览器。
- 内容最大宽度为约 4xl，窄屏下保持左右安全边距。
- 双列内容在移动端降为单列。
- 兴趣卡片在移动端为两列，在中等及以上屏幕为五列。
- 文字、卡片和固定底栏不得造成横向滚动或遮挡主要内容。
- 当前版本依赖 Google Fonts；网络不可用时使用系统回退字体。

## 10. 部署与运行要求

产品由两个运行单元组成：

1. 静态前端：构建产物位于 `artifacts/portfolio/dist/public`。
2. Node.js API：构建入口为 `artifacts/api-server/dist/index.mjs`。

生产环境必须满足：

- 域名根路径及静态资源由前端服务；
- `/api/*` 同域转发至 API 服务；
- SPA 未知前端路径回退到 `index.html`；
- API 配置 `PORT` 和 `DATABASE_URL`；
- 需要邮件通知时配置 `SMTP_USER`、`SMTP_PASS` 和可选的 `NOTIFY_EMAIL`；
- PostgreSQL 已创建 `site_stats` 和 `messages` 表；
- `/api/healthz` 可被部署平台访问。

## 11. 当前已知限制与上线前要求

以下事项尚未在当前代码中实现，不应视为已交付能力。

### P0：正式公开前必须确认

- 为留言接口增加姓名与内容长度限制，并在前后端保持一致。
- 为留言、点赞和访问接口增加基础限流，防止机器人滥用。
- 修复统计量“先读后写”的并发覆盖风险，改为数据库原子自增。
- 对邮件 HTML 中的访客输入做转义，防止内容注入。
- 确认完整手机号和邮箱保留在公开前端源码中是否可接受；如不可接受，应移至服务端或删除。
- 限制生产环境 CORS 来源，避免默认允许任意来源。

### P1：体验与可访问性

- API 失败时提供明确的加载失败、留言失败和重试反馈。
- 留言弹窗增加键盘焦点管理、Esc 关闭和无障碍标签。
- 表单使用显式 `label` 与输入框关联。
- 支持 `prefers-reduced-motion`，为减少动态效果的用户停用持续动画。
- 为语言切换、页签和弹窗补充必要的 ARIA 状态。

### P2：后续增强

- 增加 SEO 描述、规范链接、结构化数据和更完整的社交分享元信息。
- 可选增加 PDF 简历下载与独立项目详情。
- 可选增加隐私友好的访问分析。
- 可选增加内容管理方式，避免每次更新履历都修改代码。

## 12. V1 发布验收

正式发布前至少完成以下检查：

- `pnpm install --frozen-lockfile` 成功；
- `pnpm run build` 成功；
- 首页在目标域名返回 200，静态资源无 404；
- 五个内容页签在中文和英文下均可访问；
- 桌面端和移动端布局无明显溢出；
- `/api/healthz` 返回 200；
- 首次访问后访问量正确增加；
- 同一浏览器正常流程中仅可点赞一次；
- 留言可成功入库；
- 配置 SMTP 后可收到留言通知邮件；
- API 异常时页面不会白屏；
- P0 上线要求已完成或由产品所有者明确接受风险。

## 13. 产品指标建议

当前代码仅记录累计访问次数和累计点赞次数。上线后可关注：

- 网站可用性与 API 健康状态；
- 累计访问次数；
- 累计点赞次数；
- 留言提交数与邮件通知成功率；
- 移动端与桌面端访问占比（需后续引入分析能力）。

不建议在未建立独立访客识别和隐私规则前，将当前访问次数直接解释为 UV。
